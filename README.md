# LLM Evaluation Tutorial: DeepEval & OpenEvals (LangChain)

This project is a complete developer tutorial designed to demonstrate how to build a Retrieval-Augmented Generation (RAG) customer support bot and automatically evaluate its performance using two popular LLM evaluation frameworks: **DeepEval** (by Confident AI) and **OpenEvals** (by LangChain).

---

## 💡 What We Are Showing in This Project (Meeting Guide)
*Use this guide to explain the project in your meeting:*

1. **The Problem:** When building LLM applications (like support chatbots), we cannot use traditional software unit tests (e.g., `assert response == "expected_text"`) because LLM output wording is dynamic and changes slightly on every run.
2. **The Solution:** We use **LLM-as-a-Judge** frameworks. These frameworks use a powerful, separate LLM (e.g., Llama-3.3 on Groq) configured with a strict evaluation rubric to judge the bot's outputs based on factual accuracy, relevance, and lack of hallucination.
3. **The Implementation:** We created a customer support bot for a fictional health app called "FitLife" and ran two specific test scenarios to show how both frameworks analyze the responses:
   * **Test Case 1 (Happy Path):** The user asks a question about premium subscriptions. The bot retrieves the correct documentation and answers perfectly. Both frameworks give it perfect scores.
   * **Test Case 2 (Refusal/Boundary Path):** The user asks the bot to write a custom Python script. The bot does *not* have this capability or information. To prevent hallucinations, the bot politely refuses: *"I am sorry, but I do not have that information."* 
     * **DeepEval** flags this with low **Answer Relevancy** (`0.50` because it didn't give the user a script) but perfect **Faithfulness** (`1.00` because it did not make up a fake script).
     * **OpenEvals** marks it as **Incorrect** (`False`) but **Not Hallucinating** (`True`).
     * This shows how evaluation metrics let developers isolate whether a failure is due to a lack of data (low relevance) or a hallucination problem (low faithfulness).

## 🔍 Parameters Explained

LLM evaluation metrics compare different text inputs and outputs to determine quality. Below is the mapping and explanation of the parameters used in both frameworks:

| Parameter (DeepEval) | Parameter (OpenEvals) | What it Means | Why it matters |
| :--- | :--- | :--- | :--- |
| **`input`** | **`inputs`** | The user's query or prompt sent to the bot (e.g., *"How much does premium cost?"*). | Used to evaluate if the bot's response actually answers the user's question. |
| **`actual_output`** | **`outputs`** | The raw text response generated by your bot. | This is what the judge LLM is evaluating for quality, tone, and accuracy. |
| **`retrieval_context`** | **`context`** | The factual reference documents retrieved from your database or knowledge base. | The judge checks the bot's output *against this context* to ensure the bot did not make things up (hallucination). |
| **`expected_output`** | **`reference_outputs`** | The "ground truth" or ideal answer written by a human. | Used by correctness metrics to see if the bot's response aligns with the desired answer key. |

## ⚖️ DeepEval vs. OpenEvals: Technical Breakdown

### 🎯 DeepEval (Confident AI)
* **Testing Philosophy:** Built for LLM unit testing. It is designed to integrate directly with `pytest` so you can run evaluations as part of a CI/CD build pipeline.
* **Custom Models:** Requires subclassing `DeepEvalBaseLLM` and overriding `generate`/`a_generate` methods to wrap API calls (we wrapped your Groq client).
* **Key Strengths:** Great for tracking metrics over time, generating rich visual terminal tables, and continuous integration testing.

### 🚀 OpenEvals (LangChain)
* **Testing Philosophy:** Designed as functional wrappers for fast developer checks. It integrates natively with LangChain workflows and LangSmith for tracking runs.
* **Custom Models:** Extremely simple to customize; accepts LangChain chat model wrappers (like `ChatGroq`) directly without writing custom wrapper classes.
* **Key Strengths:** Seamless fit if you are already using LangChain/LangGraph, lightweight, and logs evaluation feedback automatically to LangSmith.

---

> [!WARNING]
> ### ⚠️ Key Polarity Difference (A Common Gotcha)
> Keep in mind how the two frameworks score hallucinations:
> * **DeepEval (Faithfulness):** Score of `1.00` is **excellent** (100% faithful, zero hallucinations).
> * **OpenEvals (Hallucination):** Score of `True` means **passed** (meaning the response meets the rubric of being *without* hallucinations). Always read the underlying rubric/prompt so you do not confuse the polarity of boolean results!
